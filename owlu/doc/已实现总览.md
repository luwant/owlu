# 已实现总览（截至 2026-03-13）

## 架构概览

OWLU 当前已经形成一条可运行的三模块闭环：

```text
Discovery --MatchResult--> Writer --promoted_labels / LTCE samples--> Absorption
    ^                                                                  |
    |---------------------- updated label inventory / LTCE state -------|
```

| 模块 | 职责 | 当前状态 | 文档 |
|------|------|------|------|
| Candidate Discovery | 从低置信样本中发现候选短语，并决定 merge / novel / hold | 已实现并通过真实 API 测试 | [candidate_discovery.md](candidate_discovery.md) |
| Ontology-Constrained Writer | 跨文档聚类、候选晋升、持久化、训练样本导出 | 已实现并形成 slow-sync 数据闭环 | [ontology_writer.md](ontology_writer.md) |
| Prototype Absorption | 将别名与新标签吸收到 LTCE 表示空间与分类头 | `fast_sync` 已验证，`slow_sync` 已接通真实 LTCE | [prototype_absorption.md](prototype_absorption.md) |

---

## 本次新增闭环能力

本轮最新改动主要补齐了 Writer 到 slow sync 之间缺的那一段数据链路：

1. `LabelBank` 不再只按归一化短语精确聚类，而是支持基于 dense encoder 的语义聚类。
2. `ProtoLabelCluster` 持久化保存簇质心，Writer 重启后仍能延续相同的聚类空间。
3. `OntologyWriter.ingest_with_document(...)` 可以在写入候选短语时，同时把原始文档和标签证据落到 SQLite。
4. 候选簇晋升为正式标签后，关联样本会自动从 `pending` 变成 `approved`。
5. `OntologyWriter` 现在可以直接：
   - 统计某个标签已有多少已批准正例
   - 判断哪些 promoted label 已满足 slow sync 的最小样本数
   - 导出 `LtceTextSample`，直接喂给 `ltce_bridge.py` 构造 expanded loader

这意味着当前链路已经不是“Writer 只产出 label id”，而是可以稳定产出 slow sync 所需的文本监督样本。

## Module 1: Candidate Discovery

Discovery 负责从 LTCE 低置信样本中抽取候选短语，并对每个短语输出 `MatchResult`：

- `LtceGate`
  - 基于 `raw_max_prob` 和阈值决定是否调用 LLM
- `LLMPhraseGenerator`
  - DeepSeek 兼容调用
  - JSON 容错
  - 多次采样聚合 agreement
- `SemanticMatcher`
  - 支持稀疏和 dense 两种匹配模式
- `CandidateDiscovery`
  - 将 gate -> extract -> match 串成统一入口

输出为 `list[MatchResult]`，作为 Writer 的唯一上游契约。

---

## Module 2: Ontology-Constrained Writer

Writer 现在承担三类职责：

1. 维护标签和候选簇状态
2. 对候选簇做晋升与约束校验
3. 保存文档证据并向 slow sync 导出训练样本

当前已实现的关键点：

- `LabelBank`
  - `hold / candidate / promoted` 三态管理
  - 语义聚类、簇质心更新、代表短语选择
  - 四门控晋升：`freq`、`source_docs`、`agreement`、`semantic_distance`
- `OntologyConstraintChecker`
  - 命名格式、父类存在、重复标签约束
- `LabelBankStore`
  - SQLite 持久化 label / alias / cluster / metadata
  - 文档表 `documents`
  - 标签证据表 `label_examples`
- `OntologyWriter`
  - `ingest`
  - `ingest_with_document`
  - `promote`
  - `count_label_examples`
  - `get_slow_sync_ready_labels`
  - `export_ltce_samples`

Writer 已经可以直接产出 `LtceTextSample`，不需要在 Absorption 之前再补一层手工脚本。

---

## Module 3: Prototype Absorption

Absorption 当前分为两条路径：

- `fast_sync`
  - 标签数不变
  - 用别名语义更新现有 `label_embeddings / label_prototypes`
  - 重新标定阈值
- `slow_sync`
  - 标签数从 `L` 扩到 `L'`
  - 扩展 `classifier[-1]`
  - 用 Writer 导出的文本监督样本做增量训练
  - 在 expanded label space 上重新标定阈值

`ltce_bridge.py` 已经能加载真实 `Label-gen` LTCE 运行时，并基于 `LtceTextSample` 构造 expanded loader。

## 当前判断

从代码和测试结果看，这一轮改动已经完整走通，未发现阻断性问题。当前剩余的现实约束不是实现缺失，而是数据规模：

- 新标签能否真正提升 slow sync 效果，仍取决于 Writer 累积到的高质量正例文本数量。
- 仓库内还没有固定化的 AAPD slow-sync 指标实验报告，但运行链路本身已经闭环。
